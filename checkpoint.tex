\section{Data generation}

For generating the training data, a Mario Brothers emulator \footnote{\url{https://github.com/Kautenja/gym-super-mario-bros}} was used, as the Gymnasium framework did not have the NES game we were looking for. For controlling the game while generating the data, 3 strategies were originally used. A fully random strategy, with a uniform distribution of all the provided \texttt{SIMPLE\_MOVEMENT} actions and some action stickieness over several frames. The second strategy relyed on a pre-traing AI\footnote{\url{https://github.com/yumouwei/super-mario-bros-reinforcement-learning}} with the third being a combination, with random length few second periods of one or the other.

This combination lead to a very good coverage of the first level of the game, see figure \ref{fig:level_coverage}.

The first attempt training mix (seen in these figures) was 1000 runs of combined, 100 of fully random and only one run off the optimal AI policy, given that was pixel perfect repeatable. This did not turn out to be and optimal mix, along with another issue not entirely visible from the validation images shown here.

The \texttt{SIMPLE\_MOVEMENT} control mechanism provided by the emulator does allow for movement \texttt{left} and \texttt{right}, but only has combinations with the \texttt{A} and \texttt{B} keys for the \texttt{right}, so 4 of the possible 7 actions move the game character forward. This was also exasperated by the AI strategy, which used these actions even more than the uniform random distribution. This resulted in our original training dataset of 665 816 images only containing 5,42\% \texttt{left} actions. As a consequence of these inbalances, our first round of models continued moving the character forward even without user input.

In our enhanced dataset, this was increased to 17,05\% while also increasing the number of frames where no action was taken from 14,81\% to 50,05\%. The impact of this change on our second generation of model checkpoints is not yet known.

\section{Interactive}

Given our reliance on high-power cloud infastructure for training and inference we have created a client-server type application for trying out the model interactively, like a real game. Because this is just for testing purposes, it is in the form of a portable python GUI application which initiates an SSH session with our remote VPS, starts the inference process there and then streams JPEG encoded images back while user actions are streamed to the server.

For this inference mode on the GameNGen architecture model, only 3 diffusion steps are used. We have also employed a handful of optimizations beyond the latent image buffer used in the base model (see \ref{gamengen}). One of these optimalizations is using 16-bit floating point values, over the original 32-bit. Even tho a latent buffer is used and only the output image needs to be run through the VAE for display, this task was offloaded to a second GPU along with encoding the image into JPEG for transport, keeping the primary running the UNet continuously. This increased FPS by about 12\%. As of the writing of this report we are achiving around 22,5 FPS on a pair of Nvidia RTX PRO 6000 Blackwell GPUs while only utilizing them to about 50\% and 10\% respectively. This results in the relatively tame powerdraw of only 300W combined, about half of the TDP of one of these GPUs.

Currently we do not have a remote iteractive application of the ControlNet based model.
